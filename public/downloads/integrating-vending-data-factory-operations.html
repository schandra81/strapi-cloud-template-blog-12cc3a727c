Manufacturing Vending Data Integration Pack

Use case framework, integration patterns, operator capability profile, implementation roadmap

This pack helps facility, IT, and operations leaders evaluate how vending telemetry should integrate with factory data systems (MES, CMMS, ERP, BI). It frames operator capability, lists the standards-aligned integration patterns most plants use, and gives a 90-day implementation roadmap. The frameworks are equipment- and operator-agnostic.

2. Integration patterns — pick by data freshness need

Vending data is rarely real-time-critical (unlike line equipment). Most plants land on a near-real-time pull every 15-30 minutes for dashboards plus a daily batch for ERP / BI.

Pattern A — Operator portal only
No integration. Facility logs into operator portal for QBR. Lowest cost. Acceptable for small (1-3 machine) deployments.
Pattern B — Daily CSV / SFTP batch
Operator emits daily CSV (sales, stockouts, uptime, refunds) to a facility SFTP. Loaded into BI by IT. Acceptable for QBR-cadence reporting.
Pattern C — REST API pull (NAMA DEX / EVA-DTS aligned)
Operator exposes REST API or webhook. Facility pulls every 15-30 minutes. Most common pattern for plants with active facility dashboards.
Pattern D — Direct MES / CMMS event hook
Temperature excursion or offline event posts to facility event bus (Kafka / Azure Event Grid / AWS EventBridge). Triggers CMMS ticket. Highest integration cost; used by plants with cold-chain or GMP requirements.

3. Standards + protocol reference

Vending telemetry uses a small set of well-established standards. RFP should require operator to support at least one machine-data standard plus a documented portal API.

DEX (Direct Exchange) — UCS spec
The bedrock per-SKU sales / inventory protocol every modern vending machine emits. Operator pulls DEX from each machine via cellular modem; this is what every dashboard, planogram, and stockout report rolls up from.
EVA-DTS (European Vending Association — Data Transfer Standard)
Successor / European cousin of DEX. Operator telemetry stacks (Nayax, Cantaloupe) export EVA-DTS for cross-platform compatibility.
MDB (Multi-Drop Bus / Internal Communication Protocol)
Inside-the-machine protocol between cashless reader, bill validator, coin mech, and main board. Relevant for RFP because it determines which payment + telemetry hardware can be retrofit to existing machines.
PCI DSS v4.0
All cashless transactions must run on PCI DSS v4.0-validated infrastructure. RFP requires AOC (Attestation of Compliance) on file.
ISO 27001 (operator IT) + SOC 2 Type II
Recommended for any operator that exposes a portal or API. Plants under corporate IT governance often require both.

5. Security + IT governance checklist

  • Operator cellular modem is on a separate APN — not on plant Wi-Fi or plant OT network.
  • If operator requires plant network access for any reason, segment via dedicated VLAN with deny-by-default firewall rules.
  • PCI DSS AOC on file; reviewed annually.
  • Operator portal access uses SSO (SAML / OIDC) where available; otherwise enforced MFA + least-privilege user roles.
  • API credentials rotated quarterly; stored in plant secret manager (HashiCorp Vault / Azure Key Vault / AWS Secrets Manager).
  • Webhook endpoints validate signed payloads + source IP allowlist.
  • Data classification reviewed — vending sales data typically internal-only; no PII beyond cashless transaction hashes.

6. Implementation roadmap — 90 days

  1. Days 0-15 — discovery + scoping

    Workshop with facility + IT + operator. Identify the 3-5 use cases worth integrating. Pick integration pattern (B / C / D). Document data flow.

  2. Days 15-30 — operator portal access + dashboards

    Stand up operator-portal SSO. Build initial QBR dashboard from portal exports. Confirms data quality before any code is written.

  3. Days 30-60 — API or SFTP integration

    IT integrates operator API (Pattern C) or SFTP feed (Pattern B). Lands data in plant BI / data lake. Schema validated; backfill 60-90 days.

  4. Days 60-75 — alerting + CMMS hook (if Pattern D)

    Wire offline + temperature events into facility event bus and CMMS. Run live drills (pull modem; trigger event; confirm CMMS ticket).

  5. Days 75-90 — review + handover

    QBR pack rebuilt from integrated data. Document runbook. Transition to steady-state operation. Schedule quarterly integration health check.

7. Quarterly business review template

Once integration is live, the QBR pack should be auto-generated from the integrated data — not assembled manually each quarter.

  • Uptime % per machine, per month (rolling 12 months)
  • Stockouts: count, MTTR, top-5 SKUs by stockout rate
  • Per-shift velocity report (shift-1 / shift-2 / shift-3)
  • Refrigeration excursions: count, duration, resolution time
  • Energy: kWh per machine per month, trend vs. baseline
  • Service SLA adherence by tier (Tier 1 / 2 / 3)
  • Refund rate + reason codes
  • Cashless mix % and revenue share reconciliation
