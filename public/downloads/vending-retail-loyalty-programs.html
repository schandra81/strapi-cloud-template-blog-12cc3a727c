Retail Vending Loyalty Integration Playbook

Eight integration patterns, API requirements, data-sharing agreement template

Retail vending placements that integrate with the host's loyalty program convert 15-30% higher ticket and 2-5x higher member share than unconnected machines. This playbook covers the eight integration patterns, the API requirements they each impose, the data-sharing agreement structure, and the operator-side technical baseline.

2. API requirements by pattern

Each pattern imposes specific API endpoints, authentication, rate-limit, and latency expectations on the loyalty platform. Operator-side telemetry (Cantaloupe, Nayax, 365 Retail Markets) supplies the machine side.

  1. Member lookup endpoint

    GET /members/{id} returning member tier, status, opt-in flags. <500ms p95 latency. Used by P1-P8.

  2. Points read/write

    GET + POST /members/{id}/points with idempotency-key header to prevent double-credit on retry. <1s p95 latency. Used by P2, P3.

  3. Transaction authorization

    POST /transactions with member + amount + item-set; returns approval token + adjusted price. <2s p95 latency. Used by P3-P5.

  4. Pricing API

    GET /pricing/{sku}?member={id} returning effective price. Cached on-machine for offline operation with periodic refresh. Used by P4.

  5. Wallet balance + debit

    GET /wallet/{id}/balance and POST /wallet/{id}/debit with two-phase commit (authorize + capture). Used by P5.

  6. Coupon redeem

    POST /coupons/{code}/redeem with member-id, sku, amount. Idempotency required. Used by P6.

  7. Recommendation feed

    GET /recommendations/{id}?context=vending returning ranked SKU list. Cached on-machine; refresh hourly. Used by P7.

  8. Entitlement check

    GET /entitlements/{id}?sku= returning allowed / not-allowed + remaining-balance. <1s p95 latency. Used by P8.

3. Operator-side technical baseline

Telemetry platform
Cantaloupe Seed Pro, Nayax VPOS Touch, 365 Retail Markets ConnectedFresh, or Crane Streamware. All offer REST + webhook integration for transaction events.
Payment processor
Stripe, USAePay, or operator-bundled (USA Technologies, Cantaloupe Connect). Tokenization must comply with PCI DSS v4 SAQ B-IP.
On-machine UI
Class P/A equipment (USelect VM850, Crane Media, AiFi) supports custom UI overlay for loyalty branding. Class S equipment limited to peripheral QR reader for P1.
Offline behavior
On-machine cache (pricing, member status, recommendation feed) refreshes hourly; transactions queue when offline and reconcile on reconnect.

4. Data-sharing agreement — template structure

Loyalty integration requires a written data-sharing agreement between the host (loyalty owner) and the operator (transaction collector). Use the structure below as the baseline. Add CCPA / CPRA + state-specific privacy provisions per the operating jurisdictions.

  • Parties + purpose — Host as data controller; operator as data processor under GDPR / CCPA terminology
  • Data fields shared — member ID, transaction amount, SKU, timestamp, location. NOT PII beyond member ID unless explicitly authorized
  • Permitted use — solely for loyalty program operation + reporting back to host; no operator-side resale or secondary use
  • Security — TLS 1.2+ in transit; AES-256 at rest; PCI DSS v4 attestation; SOC 2 Type II report annually
  • Retention — operator retains 18 months for reconciliation; deletes on contract end + 90 days
  • Breach notification — operator notifies host within 24 hr of suspected breach; coordinates with host on regulator + customer notice
  • Audit rights — host may audit operator security + data-handling annually; 30-day notice
  • CCPA / CPRA + state privacy law compliance — operator acknowledges service-provider obligations; host retains controller obligations
  • Termination + return — on contract end, operator returns or destroys all member data; certifies in writing

5. Implementation roadmap

  1. Phase 1 — P1 identity-only pilot

    Weeks 1-4. QR scan + transaction logging on 5-10 machines. Validate API latency, error rate, and reconciliation pipeline. Decision gate before broader rollout.

  2. Phase 2 — P2/P3 points earn + redeem

    Weeks 5-12. Add points earn on all transactions; add redeem on subset of SKUs. Iterate on UI + edge-case handling (refund, partial dispense).

  3. Phase 3 — P4 member pricing or P5 wallet

    Weeks 13-26. One of pricing or wallet, not both — choose based on member behavior data from Phase 2.

  4. Phase 4 — P6-P8 advanced

    Months 7-12. Coupon, recommendation, or subscription based on roadmap priority. Each is its own 8-12 week project.
