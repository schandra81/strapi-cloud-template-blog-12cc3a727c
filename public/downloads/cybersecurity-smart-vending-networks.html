Smart Vending Cybersecurity RFP Scorecard

Compliance attestations, MFA, breach SLA, audit log, pentest, and disclosure program

Smart vending equipment is a connected IoT endpoint inside the host's facility. Treat it like any other IoT endpoint. This pack codifies the cybersecurity attestations, controls, and disclosures the host should require from any operator running cellular-connected or campus-connected vending equipment. It maps to PCI DSS v4.0, NIST SP 800-53 / 800-171 control families, ISO/IEC 27001, NIST IR 8259 (IoT baseline), and the FTC Safeguards Rule where personal data is in scope.

1. Compliance attestations the operator must produce

PCI DSS v4.0 AOC
Current Attestation of Compliance dated within the last 12 months. SAQ type appropriate to architecture (commonly SAQ-B-IP or SAQ-P2PE).
SOC 2 Type II
From operator's cloud / settlement environment, scope including Security + Availability + Confidentiality. Report dated within last 12 months.
ISO/IEC 27001 (where applicable)
For operators serving regulated industries (healthcare, finance, government), current ISO/IEC 27001 certification preferred.
NIST SP 800-171 (federal placements)
For DoD / federal placements, CMMC Level 2 or equivalent NIST 800-171 self-assessment.
FTC Safeguards Rule (16 CFR 314)
Where operator collects + stores non-public personal information (NPI), documented information security program with designated qualified individual.

2. Identity, access, and MFA

Account compromise on the operator's back-office is the most common path to fleet-wide tampering. MFA + least-privilege are the controls that block it.

  • All operator administrative access to the back-office (planogram, pricing, telemetry) requires phishing-resistant MFA (FIDO2 / WebAuthn preferred; TOTP minimum). SMS OTP alone is insufficient.
  • Role-based access control with documented role catalog. Least-privilege enforced; route drivers cannot reach pricing or telemetry config.
  • Joiner/mover/leaver process — terminated personnel access revoked within 24 hours.
  • Service-account credentials rotated annually or on personnel change; no shared accounts.
  • Privileged-access management (PAM) for production access; session recording on admin actions.

3. Device + network controls (IoT baseline per NIST IR 8259)

  • Each machine has a unique device identity (no shared default credentials). Default credentials changed at provisioning.
  • Mutual TLS or device-certificate authentication between machine and back-office. Plaintext HTTP prohibited.
  • Cellular-modem firmware tracked; security updates applied within 30 days of vendor release for high-severity advisories.
  • Tamper detection — door-open events and physical-tamper alerts forwarded to operator SOC and host on request.
  • Network segmentation — campus-WiFi-connected machines on a dedicated VLAN with east-west restriction to back-office endpoints only.
  • No inbound public internet exposure on the machine; all traffic egress-initiated to operator back-office over TLS.

4. Payment-data architecture (PCI DSS v4.0)

P2PE or PCI-listed payment terminal
Card-present data encrypted at the reader; operator infrastructure outside the cardholder data environment (CDE) where feasible. Solution listed on PCI SSC website.
No card-data persistence
PAN, CVV, full magnetic stripe data, and PIN block must not persist on operator infrastructure. Tokenization for any rebill / loyalty linkage.
Network segmentation of CDE
If CDE present, segmented + scoped per PCI DSS requirement 1.x; segmentation testing annual.
Annual penetration test
External + internal pentest annual and after material change. Findings tracked to remediation.

6. Logging, monitoring, and audit-log access

  • Authentication, authorization, and administrative-action logs retained ≥ 12 months. Payment-system logs per PCI DSS 10.x (≥ 1 year, ≥ 90 days online).
  • Security event monitoring 24/7 — operator SOC or contracted MSSP. SIEM coverage on back-office + payment environment.
  • Host has right to request audit-log extracts for any machine within 5 business days, subject to PII/PCI redaction.
  • Annual third-party audit (SOC 2 Type II) findings shared with host on request under NDA.

7. Coordinated vulnerability disclosure program

Public disclosure channel
Operator publishes a security.txt / responsible-disclosure page with contact email, PGP key, and safe-harbor scope.
Triage SLA
Acknowledge within 5 business days; severity assignment within 10. CVSS v3.1 scoring stated.
Remediation SLA
Critical: ≤ 30 days. High: ≤ 60 days. Medium: ≤ 90 days.
Bounty / safe harbor
Documented safe-harbor language for good-faith researchers; bounty program optional but preferred.

8. Data-protection + privacy alignment

  • Data Processing Agreement (DPA) signed; sub-processor list disclosed and update notice required.
  • Data residency stated (typically US-based for US placements); international transfer mechanism documented if applicable.
  • Member-data minimization — operator stores employee ID + balance only on closed-loop; no SSN, no government ID, no compensation.
  • FTC Safeguards Rule alignment where NPI is in scope; qualified individual designated; risk assessment performed.
  • Right to audit — host (or representative) may audit operator security program annually with reasonable notice; on incident, immediately.

10. Pre-signing red flags

  • Operator cannot produce a current PCI AOC or SOC 2 Type II report.
  • Operator admins log in with SMS-only MFA or password-only.
  • Telemetry traffic uses HTTP or shared default credentials.
  • Operator refuses incident-notification SLA in writing.
  • No documented coordinated vulnerability disclosure program.
  • DPA does not include sub-processor disclosure or audit right.
