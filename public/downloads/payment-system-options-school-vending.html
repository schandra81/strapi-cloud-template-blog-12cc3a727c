School Vending Payment Playbook

K-12 prepaid integration, higher-ed campus badge, compliance framework, DPA structure

School vending payment splits into K-12 (parent-funded prepaid accounts + cashless cards) and higher-ed (campus badge + meal plan + dining dollars). Both add a privacy + compliance layer — FERPA, COPPA, PCI DSS v4, state child-data laws — that consumer vending does not face. This playbook codifies the patterns, the compliance framework, and the data-processing-agreement structure that procurement reviewers expect.

2. K-12 prepaid integration

Parent-funded prepaid platforms (MySchoolBucks, SchoolCafé, LINQ Connect, TITAN) integrate with the cafeteria POS first and extend to vending second. Operator-side telemetry (Cantaloupe, Nayax) reads a student PIN or card swipe and authorizes against the prepaid balance via API.

  1. Identification mechanism

    Student PIN at on-machine keypad, mag-stripe student ID swipe, or barcoded student ID scan. Some districts use proximity card or app-based QR.

  2. Authorization flow

    Operator POS → prepaid platform API → balance check → debit + transaction record. <2s p95 latency target. Idempotency-key required to prevent double-debit on retry.

  3. Reconciliation

    Daily settlement from prepaid platform to operator via ACH; per-transaction record retained by both for audit. Variance investigation if daily settlement differs > 0.05% from operator telemetry.

  4. Refund flow

    Refund credits prepaid balance within 24 hr; refund report to district nutrition services + parent visibility through prepaid app.

3. Higher-ed campus badge + meal plan

  1. Campus card platforms

    Blackboard Transact (formerly Blackboard), CBORD Odyssey + Get, Atrium, TouchNet OneCard. Each integrates with operator POS via REST or proprietary card-network protocol.

  2. Tender types

    (a) Dining dollars / flex dollars — closed-loop currency. (b) Meal-plan equivalency — meal swipe → $5-$10 vending equivalent. (c) Bonus bucks — sponsor/wellness-funded. (d) Direct campus-card debit.

  3. Late-night + dorm placement

    Campus card-only access after dining hall closure; common pattern in dorm + library 24/7 placements. Operator POS recognizes card status (active / hold / inactive).

  4. Cross-campus portability

    Multi-campus systems may allow campus card to spend at any system campus; operator POS validates against the system-wide card network.

4. Compliance framework

School vending payment crosses four federal regimes plus state child-data laws. Operator + district share responsibility and that division must be in writing.

FERPA — Family Educational Rights and Privacy Act (20 USC §1232g)
Student records (including transaction records tied to a student) are protected. Operator acts as 'school official' with legitimate educational interest under 34 CFR §99.31(a)(1)(i)(B). Written agreement required defining permitted use + retention + deletion.
COPPA — Children's Online Privacy Protection Act (15 USC §6501-§6506)
Applies to data collection from children under 13. K-12 payment platforms generally fall under COPPA; operator must avoid collecting any data beyond what's needed for the transaction.
PCI DSS v4 — Payment Card Industry Data Security Standard
Operator merchant level (typically Level 3 or 4 for vending). SAQ B-IP attestation for tokenized cashless; SAQ D for stored cardholder data. Annual attestation required.
State child-data laws
CA AB-1584, CT Public Act 16-189, NY Ed Law §2-d, IL SOPPA, CO HB-1135, TN HB-1931 — vendor + DPA requirements. Operator must comply with each state where district operates.
USDA child-nutrition + Smart Snacks
7 CFR §210.11 + §220.12. Where vending integrates with cafeteria lunch program, USDA audit reach extends to vending transaction records during school day.

5. Data Processing Agreement — required structure

  • Parties + purpose — district as controller; operator + prepaid platform as processors. Define data flow.
  • Data fields — student identifier (district-assigned ID, not SSN), grade, transaction amount, SKU, timestamp, location. NO direct PII beyond ID unless explicitly authorized.
  • FERPA designation — operator as 'school official' with legitimate educational interest; written designation per 34 CFR §99.31(a)(1)(i)(B)
  • COPPA representations — operator confirms no direct collection from children under 13; relies on district as COPPA-authorized agent
  • Security — TLS 1.2+ in transit; AES-256 at rest; PCI DSS v4 attestation; SOC 2 Type II annual report; encryption-key management
  • Retention — operator retains 18 months for reconciliation; deletes within 90 days of contract end + provides certificate of destruction
  • Breach notification — operator notifies district within 24 hr of suspected breach; coordinates with district on parent + regulator + AG notice per state law
  • Audit rights — district may audit operator annually; right to require third-party penetration test results
  • Subcontractor flow-down — operator binds all subprocessors (telemetry, payment processor, host) to equivalent terms
  • State-specific addenda — append CA AB-1584, NY Ed Law 2-d, IL SOPPA, CO HB-1135 riders per operating jurisdictions

6. Operational support structure

Restock cadence
Telemetry-driven; no change from base. Reconciliation pipeline runs daily; variance > 0.05% triggers investigation.
Service SLA — Tier 1
Offline, cold-chain excursion, payment failure — <24 hr ack + <24 hr resolution. Payment integration failure (operator side) — <4 hr ack + <8 hr resolution.
Prepaid platform SLA
≥ 99.9% uptime; <2s p95 latency; <0.1% error rate. Operator falls back to cashless on prepaid API failure; FERPA-compliant transaction queue on operator side.
Refund flow
Refund credits prepaid balance or campus card within 24 hr. Cash refund handled by school front office per district policy.
Annual review
DPA + PCI DSS attestation + SOC 2 + insurance certificate reviewed at contract anniversary; renewed with current documents on file.
