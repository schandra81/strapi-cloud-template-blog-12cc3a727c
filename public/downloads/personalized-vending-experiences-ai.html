Personalized Vending Pilot Framework

Placement fit, recommendation models, privacy architecture, opt-in design, and pilot measurement

Personalized vending (AI-driven recommendations, member-specific pricing, dietary-preference surfacing) only earns its place where placement, privacy architecture, and opt-in design are all in good shape. This pack defines a 90-day pilot framework with measurable activation, repeat, and consent metrics. It maps to FTC Section 5 (unfair/deceptive practices), CCPA / CPRA where applicable, GDPR where applicable, ADA Section 309.4, and FALCPA allergen labeling.

1. Placement-fit screen — where personalization pays back

Personalization adds operating complexity (consent management, model maintenance, A/B testing infrastructure). It pays back only where placement supports repeat visits and identity is already known.

High-fit placements
Corporate campus with badge/SSO identity, healthcare staff areas, university residence + dining, multi-unit residential with member app, 24/7 micro-markets with closed-loop.
Marginal-fit placements
Mixed-use lobbies with anonymous walk-up traffic, transit hubs, hospitality.
Poor-fit placements
Single anonymous machine, sub-100-employee single-site office, schools subject to USDA Smart Snacks (use compliance-locked planogram, not personalization).

2. Recommendation-model design

  1. Cold-start (visit 1)

    No member history. Recommend by placement-level top-decile velocity + healthy-share floor + allergen-flag respect from member-provided allergen list (opt-in).

  2. Warm-start (visits 2-5)

    Collaborative-filter + content-based hybrid using member's prior purchases and similar-member purchases at the placement. Diversity term injected to avoid filter bubble.

  3. Steady-state (6+ visits)

    Personalized ranking on a fixed planogram; recommended SKUs surfaced first; remaining catalog browsable. Recommendation diversity audited monthly.

  4. Negative signals

    Explicit dislike (member flags 'don't recommend'), refund flag, allergen mismatch all feed back into model. Refund rate by recommended SKU is a model-quality KPI.

  5. Fairness checks

    Model output audited for systematic SKU exclusion across protected-class proxies (where any demographic is collected for benefit-eligibility reasons).

3. Privacy architecture

  • Data minimization — store employee/member ID + purchase events + opt-in preferences. Do not store name, address, government ID, payroll details, compensation, benefits-eligibility data.
  • Purpose limitation — purchase data used for personalization + operations; not sold or used for advertising; not shared with employer in member-identifiable form.
  • Retention — member-level purchase data retained for the period stated in privacy notice (commonly 12-24 months); deleted on member request.
  • Pseudonymization — model trained on pseudonymized member IDs; re-identification only at the recommendation API for the active member.
  • Data residency — primary store US-based for US placements; international transfer mechanism documented if applicable.
  • Sub-processor disclosure — list of ML platform, hosting, analytics vendors with current security attestations.

4. Opt-in design (FTC + CCPA aligned)

Personalization requires affirmative opt-in where placement triggers state-privacy-law scope (CA, CO, CT, UT, VA, TX, etc.). Even outside those scopes, opt-in is the FTC-aligned standard.

  • Clear, layered privacy notice presented at app onboarding and at first on-machine personalization moment.
  • Toggle for 'personalize recommendations' default OFF; member must enable. Toggle reversible at any time.
  • Granular consent — recommendations on, member-specific pricing on, marketing communications on — each separate.
  • Allergen-list opt-in stored separately and surfaced as a safety-tool option even if personalization is off.
  • Member dashboard shows data collected, purpose, retention, and 'delete my data' button. Deletion completed within 30 days.
  • Consent log retained for the duration of the relationship + 3 years; auditable.

6. Accessibility + safety guardrails

  • ADA Section 309.4 — recommendation interface operable with one hand and ≤ 5 lbf operating force.
  • Screen content available via audio output / accessible alternative for members with vision impairment.
  • Allergen labeling per FALCPA — recommendations respect member allergen list; if no list, generic allergen labeling visible on each SKU.
  • Healthy-share floor enforced — recommendation engine cannot rank healthy SKUs below planogram floor allocation.
  • No exclusionary pricing — member-specific pricing pilots run only against opt-in, with transparent rules; no protected-class proxies.

7. Operator + vendor diligence checklist

  • ML platform — current SOC 2 Type II; model-card documentation for the recommendation model.
  • Bias + fairness testing performed during model development; results shared with host under NDA.
  • Sub-processor list disclosed; data-processing addendum (DPA) signed.
  • Right-to-audit clause included; host or designee may audit annually with notice.
  • Model retraining cadence documented (typically monthly); change-management process for material model updates.

8. Pilot acceptance + remedies

  1. Acceptance

    All KPIs in section 5 met or exceeded over the final 30 days of the 90-day pilot. Privacy + accessibility checklists pass.

  2. Partial acceptance

    5 of 8 KPIs met; remediation plan agreed for the rest. Re-measure over a 60-day extension.

  3. Non-acceptance

    ≤ 4 KPIs met or any privacy/accessibility checklist fails. Operator removes personalization layer at own cost; baseline planogram restored; no penalty to host.

9. Pre-signing red flags

  • Operator collects PII beyond minimization standard (name + address + DOB) for personalization.
  • Opt-in default ON, or consent buried in terms of service.
  • No deletion mechanism or deletion SLA > 30 days.
  • Model-card + bias-testing documentation refused.
  • Recommendation engine can override healthy-share floor or allergen flags.
This pack is informational and not legal, privacy, or AI-governance advice. Personalization pilots involving employee data should be reviewed by HR, privacy/legal, and security teams. State-privacy-law scoping (CCPA/CPRA, CO, CT, UT, VA, TX) and federal sectoral rules where applicable (HIPAA, FERPA, COPPA) should be confirmed with counsel.
