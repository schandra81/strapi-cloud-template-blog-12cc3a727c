Custom Vending RFP Pack

5-dimension customization spec, operator scoring matrix, and contract addendum

Use this pack when the standard VMS placement doesn't fit — branded wraps, custom planograms, integrated identity, alternative form factors, or member-app experience. The pack codifies a 5-dimension customization spec, an RFP scoring matrix, and the contract addendum language that protects the host's customization investment.

1. The 5 customization dimensions

Most 'custom vending' requests reduce to one or more of these five dimensions. Specifying which dimensions matter — and which don't — saves 30-60 days in RFP cycle time.

Dimension 1 — Hardware form factor
Standard glass-front vs. micro-market vs. smart-cooler vs. locker-pickup vs. branded enclosure / wrap. Includes physical footprint, power draw, network, and ADA reach-range constraints (Section 308).
Dimension 2 — Planogram + SKU
Open product set, branded-product carve-outs, healthy-share floor (NANA 200/35/230 thresholds), allergen labeling per FALCPA, local-source sourcing requirements.
Dimension 3 — Identity + payment rails
Closed-loop badge/SSO integration, payroll deduct, subsidized pricing tiers, tier-based pricing for member loyalty programs, EMV + contactless + wallet rails.
Dimension 4 — Branding + member experience
Vinyl wrap, screen branding, app branding, on-machine signage, recommendation prompts, member dashboard, in-app survey + feedback loop.
Dimension 5 — Data + analytics
Per-SKU velocity, per-segment heatmap, A/B test framework, KPI dashboard, integration to host BI (Tableau, Power BI, Looker), data-retention + access terms.

3. Operator-capability questionnaire

  1. Hardware portfolio

    Which OEMs do you stock? Do you maintain inventory of smart coolers + micro-market kiosks + locker systems, or do you order on demand? Lead time at signing?

  2. Wrap + branding workflow

    In-house print + install or third-party? Wrap-design approval cycle (rounds, days)? Damage-replacement policy?

  3. Identity-system integrations

    Confirmed integrations to Okta, Azure AD, campus SSO, HID, Lenel, badge providers? Reference accounts using each?

  4. Closed-loop + subsidy

    Tier-based pricing engine — operator-native vs. via partner? Settlement cadence to host AP? SOC 2 Type II for the settlement environment?

  5. Data + analytics

    Native dashboard, scheduled export, or API? Schema documentation? Data-retention period? Per-member vs. aggregate access?

  6. Pilot capability

    Can you run a 90-day pilot with measurable KPIs before full deployment? What's the cost structure?

5. Pilot KPI framework

Activation rate
% of eligible members who complete first transaction in 30 days. Target: 60%+ at office; 40%+ at residential.
Repeat rate
% of activated members with 2nd transaction in 30 days. Target: 70%+.
Velocity per slot
Transactions per slot per day vs. baseline machine. Target: parity or better within 60 days.
Healthy-share consumption
% of transactions on healthy-flagged SKUs. Track against planogram allocation.
NPS + feedback
In-app survey at 30/60/90 days. NPS target +30 minimum.

6. Contract addendum language (Schedule A)

  • Schedule A enumerates each of the 5 dimensions with desired state, constraint, success criterion (per the worksheet).
  • Operator responsibilities itemized per dimension (hardware delivery + install, wrap design + production + install, integration build + test, planogram launch + maintenance, data feed delivery).
  • Host responsibilities itemized (IT/identity coordination, brand asset delivery, comms + change management).
  • Pilot acceptance criteria (KPIs above) and remedies if criteria not met (re-pilot at operator cost, scope reduction, exit at no penalty).
  • Customization cost amortization — wrap/integration cost amortized over initial term; early-exit recapture defined.
  • Data ownership — host owns transactional data; operator licensed for service + de-identified benchmarking only.
  • IP — branding assets remain host property; operator returns or destroys on termination.

8. Pre-signing red flags

  • Operator commits to all 5 dimensions but provides no comparable references.
  • Pilot proposed with no acceptance criteria — soft launch with no exit.
  • Customization cost amortized over a longer term than the initial term, locking the host.
  • Data terms grant operator perpetual rights to member-level transaction data.
  • Integration scope underdocumented — operator promises 'standard SSO' without naming IdPs + reference accounts.
