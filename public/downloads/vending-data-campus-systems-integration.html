Campus Vending Data Integration Pack

Architecture diagram, system-by-system integration spec, FERPA + IRB framework, BI dashboard templates, operator data API spec

Campus AI cooler programs emit transaction, telemetry, and (where SDK-integrated) campus-card data. This pack bundles the integration architecture, system-by-system spec, FERPA + IRB framework, BI dashboard templates, and operator data API expectations a campus IT + auxiliary services + institutional research team needs.

1. Integration architecture (text diagram)

Data flows in 4 layers: cooler edge → operator cloud → campus integration tier → campus consumers (BI, finance, dietitian, research). Each layer has a clear boundary + governance owner.

Layer 1 — cooler edge
Vision system, payment terminal, sensors, telemetry agent. Emits: anonymized transaction events, telemetry pings, sensor alerts. PCI tokenized at terminal; no PAN leaves cooler.
Layer 2 — operator cloud
Operator's SaaS platform: aggregates events, computes velocity + uptime, runs vision model, manages refund flow. Hosted in operator AWS/Azure with SOC 2 Type II posture.
Layer 3 — campus integration tier
Campus integration hub (Mulesoft, Boomi, custom .NET, or campus enterprise API gateway). Receives operator webhook + REST pulls; transforms; routes to campus consumers. Owned by Campus IT.
Layer 4a — finance + ERP
Auxiliary services GL mapping; commission reconciliation; tax + sales tax remittance. Workday Financials, Banner, PeopleSoft, or comparable.
Layer 4b — BI + analytics
Tableau, Power BI, Looker dashboards for auxiliary services + dietitian + sustainability + IR. Anonymized + aggregated only unless FERPA DPA + IRB approves identifiable use.
Layer 4c — campus card platform
Bi-directional SDK with Atrium / Blackboard / CBORD / TouchNet for tender posting + balance check.
Layer 4d — research data store
De-identified longitudinal data store for IRB-approved nutrition / food-security / behavior research.

2. System-by-system integration spec

Use the spec below for each campus system the cooler program touches. Confirm operator attestation + integration test results.

Campus card platform (one-card)
SDK / direct API certified by Atrium/Blackboard/CBORD/TouchNet; tender mapping; daily settlement file; FERPA DPA; SOC 2 in good standing.
Identity provider (IdP)
Operator admin portal SSO via Okta/Azure AD/Shibboleth; SCIM for user provisioning; MFA enforced; SAML 2.0 or OIDC.
Campus network
Cooler fleet on segmented VLAN; egress whitelist to operator + payment processor; ingress for telemetry only; no lateral path to admin networks.
ERP / finance
Monthly commission file (CSV or API) mapped to GL codes; sales tax remittance per state code; quarterly true-up.
Dietary / dining services system
Optional: dietitian audit export from operator (HHS-aligned SKU list, packaging, sourcing) to dining services planogram management.
BI + reporting
Operator data warehouse export (S3, BigQuery, Snowflake, REST API) to campus BI; cadence daily; data dictionary versioned.
Sustainability + STARS
Quarterly sustainability metrics report (local share, plant-forward, packaging, ENERGY STAR, workforce) to sustainability coordinator.
IRB / research
De-identified data store with IRB-approved access tier; longitudinal panel for food-security or behavior research; data-use agreement per research project.

3. FERPA + IRB framework

Cooler programs collect transaction data tied to campus card tender = student-ID-linked data. FERPA + (in some cases) GLBA + IRB apply.

  • Operator-signed FERPA DPA covering any student-ID data that traverses operator systems via campus card SDK
  • GLBA acknowledgment if cooler tender includes financial-aid balance or refund-disbursement integration
  • Data minimization: only required fields traverse operator; no birthdate, SSN, home address, or grade fields
  • Retention: identifiable data ≤ 90 days at operator; aggregated/anonymized longer with privacy-office sign-off
  • Re-identification ban: no joining cooler transaction data with academic or financial-aid records without IRB approval
  • Research use: separate IRB protocol required for any use beyond operations; consent + de-identification plan included
  • Disclosure log: any non-routine disclosure logged; FERPA-compliant audit trail
  • Annual privacy-office review of data flows + DPA scope

4. BI dashboard templates

Build the dashboards below in Tableau / Power BI / Looker; cadence daily or weekly. Each dashboard has a defined consumer + use case.

Operations dashboard (auxiliary services)
Per-cooler uptime, daily/weekly transactions, basket size trend, refund rate, top-15/bottom-15 SKUs, service ticket queue. Daily refresh.
Dietitian dashboard
HHS-aligned share %, sugar-sweetened beverage share, allergen labeling completeness, dietary-attestation accuracy, planogram refresh suggestions. Weekly refresh.
Sustainability dashboard
Local + regional share, plant-forward share, packaging-recyclability share, ENERGY STAR equipment %, kWh per cooler. Quarterly refresh.
Finance dashboard
Commission revenue YTD vs target, per-placement revenue trend, sales-tax remittance, refund + service-credit impact. Monthly refresh.
Security + access dashboard
Cooler offline incidents, tilt/door alerts, refund anomaly, vision-system drift, refund rate by cooler. Daily refresh.
Student wellness + experience dashboard
Survey-measured satisfaction, complaint volume, accessibility-accommodation requests. Quarterly refresh.
Research / IR dashboard (de-identified)
Aggregated longitudinal trends (consumption mix, healthy-share trend, late-night transaction pattern). IRB-approved use only.

5. Operator data API spec

Operator should expose the API spec below to campus IT for integration. Confirm endpoints, authentication, rate limits, and data dictionary.

Authentication
OAuth 2.0 client credentials; tokens scoped per consumer; MFA on operator portal for human admin.
Transaction events
Webhook POST per transaction (timestamp, cooler ID, anonymized session ID, SKU, qty, price, tender type, refund flag). REST GET supports backfill.
Telemetry events
REST GET for hourly aggregates: uptime, temperature (refrigerated), door-open count, anomaly flags. Webhook for real-time alerts.
Planogram + catalog
REST GET for current SKU catalog per cooler: SKU, name, allergens, dietary attestation, HHS-aligned flag, sourcing flags.
Refund + dispute
REST POST + GET for refund flow; status webhook for credit-back confirmation.
Commission + finance
Monthly CSV or REST GET for commission report; per-placement, per-SKU, per-tender breakdown.
Sustainability metrics
Quarterly REST GET for STARS-aligned metrics export.
Rate limits + retries
100 req/sec sustained, 1000 req/sec burst; exponential backoff guidance; idempotency keys on POSTs.

7. Implementation phases

  1. Phase 0 — discovery + DPA

    Privacy office signs FERPA DPA; IT confirms identity + network architecture; finance confirms GL mapping; sustainability confirms STARS scope.

  2. Phase 1 — operator integration

    Operator webhook + REST endpoints configured; campus integration tier built; test in dev environment; sample data exchange.

  3. Phase 2 — campus card + finance

    Campus card SDK certified; finance commission file in production; reconciliation workflow live.

  4. Phase 3 — BI dashboards

    Operations + dietitian + sustainability + finance dashboards built; user access provisioned via IdP; access tier mapped to data class.

  5. Phase 4 — research tier

    De-identified data store stood up; IRB protocols established; first research project access tier provisioned.

  6. Phase 5 — optimization

    Predictive restock model, dynamic planogram, sponsor reporting; quarterly review of integration scope + new use cases.

8. Quarterly + annual review checklist

  • Quarterly: data-flow + DPA review with privacy office
  • Quarterly: BI dashboard usage + access audit
  • Quarterly: API uptime + rate-limit incidents
  • Annually: SOC 2 + PCI + FERPA DPA refresh
  • Annually: data classification + retention schedule review
  • Annually: IRB-approved research access tier review
  • Annually: integration scope amendment if new systems added
